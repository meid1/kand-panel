# Отказоустойчивость флота (failover / балансировка / авто-хил)

Kand держит подписку живой, даже когда отдельные ноды падают.

## Мониторинг здоровья
Воркер раз в 5 минут опрашивает агент каждой ноды (mTLS `/health`). Смена состояния
online↔offline пишется в БД (`Node.online`) и шлётся владельцу в Telegram.

## Failover (в подписке — только живые ноды)
При выдаче подписки офлайн-ноды **исключаются**, если есть хотя бы одна живая — клиент не
получает мёртвые серверы. Если живых нет вовсе (например, монитор ещё не прошёл после
рестарта панели) — отдаём все ноды, чтобы не оставить клиента без VPN.
- Настройка `sub.failover` (`1` = вкл, по умолчанию; `0` = отдавать все ноды).

## Балансировка
Порядок нод в подписке детерминированно **ротируется по токену клиента** — разные клиенты
получают разный «первый» сервер, нагрузка размазывается по флоту, а приложение всё равно
видит весь список для ручного переключения.

## Авто-хил (self-heal при возврате ноды)
Когда нода возвращается в онлайн (могла перезагрузиться и потерять состояние xray), панель
**автоматически перераскатывает на неё ключи** (reconcile) — доступ восстанавливается без
ручных действий. Раздача ключей идемпотентна, повторный прогон безопасен.

## Масштабирование
Несколько нод в одном пуле = горизонтальное масштабирование: добавляете ноду одной командой,
она сразу попадает в подписки (с учётом failover/балансировки). Ноды франшиз (BYON) живут в
своём пуле (изоляция по `tenantId`).
