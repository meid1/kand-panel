# Лимит устройств: по количеству, HWID и IP

Три способа ограничить число устройств у клиента. Все опциональны.

## 1. Лимит подписок (простой)
Настройка `device.limit` (в админке → Бренд → Настройки). Ограничивает, сколько
устройств-подписок (uuid) может создать клиент. 0 = без лимита.

## 2. HWID — лимит по «железу» устройства (рекомендуется)
Приложение при запросе подписки шлёт заголовок `x-hwid` (уникальный id устройства).
Панель считает уникальные HWID клиента и не пускает новые сверх лимита.

**Как включить:** настройка `hwid.limit` (0 = выкл). При превышении новое устройство
получает 403; уже известные — работают. Клиенты, которые НЕ шлют HWID, не блокируются
(совместимость). В карточке клиента видно HWID-устройства, можно сбросить лишние.

**Какие приложения шлют HWID (инструкция клиенту):**
- ✅ **Happ**, **v2rayTun** — шлют `x-hwid` из коробки. Рекомендуйте их клиентам.
- ⏳ v2rayNG / Hiddify — поддержка HWID в разработке (следите за обновлениями).
Клиенту: установите Happ или v2rayTun, добавьте подписку — устройство привяжется автоматически.

Дополнительные заголовки (необязательные): `x-device-os`, `x-device-model`, `x-ver-os`.

## 3. IP-лимит (одновременные подключения) — на стороне ноды
У xray НЕТ нативного лимита IP на пользователя. Он делается на НОДЕ разбором логов
xray + бан лишних IP через fail2ban/iptables. Это продвинутая настройка ноды:

1. В xray-конфиге ноды включите access-log с email:
   `"log": { "access": "/var/log/xray/access.log", "loglevel": "warning" }`
   (в Kand конфиг ноды правится в админке: «⚙ конфиг»).
2. Скрипт/`fail2ban` парсит лог: считает уникальные IP на каждый `email` (это id
   устройства в Kand); при превышении N — временно банит лишние IP через iptables.
3. Готовые решения-примеры: скрипты IP-Limit для 3x-ui/Marzban (тот же принцип —
   лог xray → iptables). Их можно адаптировать под ноду Kand.

> Итог: **HWID-лимит** (панельный, привязка к устройству) — проще и рекомендуется.
> **IP-лимит** (одновременные соединения) — нодовая надстройка через логи+fail2ban.
